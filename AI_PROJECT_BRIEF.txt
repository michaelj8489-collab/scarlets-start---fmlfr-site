============================================================
1. PROJECT IDENTITY
============================================================

Project Name:
Scarlet Star Broadcasting

Short Brand Name:
Scarlet Star

Core Mission:
Scarlet Star exists to give StarMaker singers and cover artists a professional stage, a community, and a way to be discovered through live radio shows, artist pages, featured spotlights, and a 24/7 station loop.

This is not just a radio website.
This is an artist discovery platform, internet radio station, live show hub, and future integration point for Stream Suite.

Primary Brand Feeling:
Premium, cinematic, modern, dark, music-focused, community-driven.

Visual Identity:
- Black / near-black background
- Deep scarlet red as the primary accent
- Elegant, dramatic, polished
- The logo is the design anchor
- The site should feel like a premium media platform, not a generic website

Brand Keywords:
Scarlet, Star, Broadcasting, Radio, Stage, Spotlight, Artist, Live, Covers, Community, Discovery.

Primary Motto / Positioning:
"Giving cover artists a stage."

Alternative positioning phrases:
- "The home for StarMaker artists."
- "Where cover artists get heard."
- "A live stage for rising voices."
- "Your voice. Your stage. Your spotlight."

2. GOLDEN RULE
============================================================

The number one design and development question is:

"How do we make this so simple that people do not just use it, they enjoy using it and never want to go back?"

Every feature should reduce friction.

Do not build features just because they are cool.
Build features that help artists, DJs, listeners, or admins do something more easily.

Priority users:
1. Artists / singers
2. DJs / show hosts
3. Listeners
4. Admins / station staff

5. BUSINESS MODEL AND LEGAL POSITIONING GUARDRAILS
============================================================

Important rule:
Paid subscriptions are NOT payment for radio plays.

The paid membership must be presented as payment for artist services, such as:
- Dedicated artist page
- Artist bio
- Embedded media player with top StarMaker covers
- Custom artist graphics/designs
- Social media links
- Featured artist show eligibility
- Promotional spotlight
- Community visibility

Free and paid users may be heard on live shows or station programming depending on site rules.

Song plays must not be phrased as something users are directly buying.

Avoid wording like:
- "Pay $20 to get your songs played"
- "Buy radio plays"
- "Guaranteed station rotation"
- "Pay for airplay"

Use wording like:
- "Membership includes a professional artist page and promotional features."
- "Members may submit performances for consideration."
- "Selected submissions may be featured in live shows or rotation."
- "Station programming is curated by Scarlet Star staff."

Artist consent:
When artists submit songs, they must agree that they have permission to submit the recording and authorize Scarlet Star to play their vocal performance/recording on Scarlet Star programming.

The consent checkbox should be explicit.

Example consent language:
"I confirm that I am authorized to submit this performance and I grant Scarlet Star Broadcasting permission to stream, feature, and promote this submitted recording as part of Scarlet Star programming, artist features, live shows, and station-related promotional content."

There should eventually be:
- Terms of Service
- Privacy Policy
- Artist Submission Agreement
- Takedown / removal request process

21. PWA, MOBILE, AND AUTH GATE REQUIREMENTS

Scarlet Star must be built as a mobile-first responsive website AND as a Progressive Web App.

The site should feel excellent on:
- Desktop
- Laptop
- Tablet
- Mobile browser
- Installed mobile PWA app

The PWA version should use the square Scarlet Star logo as the app icon.

The site should prompt eligible users to install the app when they first visit or shortly after they engage with the site.

Important PWA note:
Browsers control the exact install behavior. On Android/Chrome/Edge, the app can show a custom install banner using the beforeinstallprompt event. On iPhone/iPad, Safari does not allow the same native prompt, so the site should show simple "Add to Home Screen" instructions for iOS users.

PWA MVP requirements:
- manifest.json or Next.js metadata manifest
- App name: Scarlet Star
- Full name: Scarlet Star Broadcasting
- Square app icon using the square Scarlet Star logo
- Theme color: deep scarlet red
- Background color: black / near-black
- Display mode: standalone
- Mobile safe-area support
- Offline fallback page or simple offline message
- Install prompt component
- iOS Add to Home Screen helper
- PWA-ready layout and icons
- Responsive navigation suitable for mobile

Future PWA features:
- Push notifications
- New chat message notifications
- Direct message notifications
- Tagged message notifications
- Show starting notifications
- Artist song going live notifications
- Featured artist alerts
- Submission status updates

Push notification button:
Eventually, authenticated users should have a native-style button in their dashboard/settings that allows them to turn push notifications on or off.

Example button text:
"Turn On Notifications"

Possible notification types:
- You were tagged in chat

Do not build the full push notification system until the core site is stable unless specifically instructed.

24. AUTH MIDDLEWARE REQUIREMENTS
============================================================

Use middleware or server-side auth checks to protect private routes.

Do not rely only on hiding links in the UI.

If a logged-out user tries to access a private route, redirect them to login or the public landing page.

Suggested behavior:
- Logged-out user visits /listen -> redirect to /login
- Logged-out user visits /artists -> redirect to /login
- Logged-out user visits /chat -> redirect to /login
- Logged-in listener visits /admin -> show unauthorized or redirect
- Logged-in artist visits /admin -> show unauthorized or redirect
- Logged-in admin visits /admin -> allow access

Role-based protection:
- Listener routes require authentication
- Artist routes require artist role or higher
- DJ routes require DJ role or admin
- Admin routes require admin role
